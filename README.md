## Databases

**Artifact:** `CRUD_Python_Module.py`  

**Description:**  
This artifact demonstrates a reusable MongoDB CRUD abstraction layer and a dashboard interface that visualizes stored data. It emphasizes database abstraction, error handling, and application integration.

**Enhancements Made:**
- Strengthened method documentation
- Added error handling and validation
- Centralized configuration
- Clarified architectural boundaries between database and presentation layers

 View CRUD Module:  
[CRUD_Python_Module.py](https://github.com/CameronSharp9402/CameronSharp9402.github.io/blob/main/(Revised)%20CRUD_Python_Module.py)

### Narrative Reflection
The ePortfolio artifact that I selected was originally developed for CS-340 Project Two, where I implemented a Python CRUD module for a MongoDB database, along with a Jupyter Notebook dashboard to visualize and interact with the data. These components were just a part of a project at that time; I have now given them a new life by polishing and upgrading them to the quality of professional software engineering, thus making them suitable for my ePortfolio. My choice of this artifact was because it can showcase a number of essential software development skills.
The CRUD module puts a spotlight on my capabilities in structuring modular code with a strong separation of concerns, hiding the database operations behind the interfaces, which are clear and well defined, thus leading to code reuse and maintaining the code base, as well. To complete this picture of an application, the dashboard demonstrates the integration of the different parts of the application, the designing of the user interface which is interactive, and finally, the conversion of the backend data into visually attractive formats that the end users can easily comprehend.
To improve the artifact, I decided to concentrate on aspects such as clarity, structure, and long, term maintainability, leaving core functionality mostly untouched. Among other changes, the code was split up so that every method fulfilled one clear responsibility. Different elements were renamed in a strictly consistent manner, error handling was made more robust, and the inline code documentation was thoroughly elaborated so that the design idea became easily graspable by others. Moreover, I clarified how the dashboard gets its data and came up with a better way of organizing callbacks and layout components. Also, I made sure to clearly distinguish between presentation and database logic.
These improvements are perfectly in line with the Module One outcomes which mainly revolve around using software engineering principles to design and assess computing solutions. This exercise has helped me to realize the value of several professional practices such as defensive programming, clear documentation, and modular system design. One of the major problems faced was figuring out how to improve the structure of tightly coupled, poorly documented, and essentially untested code while making sure that the existing working functionality was not disrupted. To solve this, I had to dissect the code very thoroughly and make changes little by little which in a way emphasized the skill of respecting legacy code while at the same time trying to get better quality. This is indeed a very useful skill in professional software development.
